# 暗号資産アルゴリズム取引システム

個人で設計・開発し、クラウドサーバー上で実際に稼働させている自動売買システムです。  
（セキュリティの観点から一部のみを公開しています。）

---

## このプロジェクトについて

「取引の仮説を立て、データで検証し、実際の市場で動かして確かめる」という一連の流れを、個人で一から構築しました。

- **稼働開始**: 2026年3月
- **取引所**: Hyperliquid（暗号資産デリバティブ）
- **実行環境**: クラウドVPS上で週4回、自動実行
- **開発規模**: 本番稼働コード 約1,700行（`production/run_donchian_hl.py`）

---

## 取引ロジック

### 全体の流れ

```
① OHLCVデータ取得
        ↓
② シグナルスコア計算（Donchian Channel）
        ↓
③ フィルタリング（3種）
        ↓
④ ロング・ショート銘柄の選定
        ↓
⑤ ポジションサイジング（ATRベース）
        ↓
⑥ 注文執行（指値 → 成行フォールバック）
```

---

### ① シグナル（Donchian Channel Score）

各銘柄に対して「現在値が過去の高値・安値レンジのどこにいるか」をスコア化します。

```
score = (現在値 - 過去10日の最安値) / (過去168日の最高値 - 過去10日の最安値) - 0.5
```

| スコア | 意味 |
|--------|------|
| +0.5に近い | 高値圏にいる → ロング候補（上昇トレンド中） |
| -0.5に近い | 安値圏にいる → ショート候補（下落トレンド中） |

「長期で高値を更新し続けている銘柄を買い、安値を更新し続けている銘柄を売る」トレンドフォロー戦略です。

---

### ② フィルタリング（3種）

シグナルだけでは不十分なため、3つのフィルタで精度を高めます。

**[A] BTCレジームフィルタ**  
BTC価格が200日移動平均を下回っている場合、相場全体がベア（下落相場）と判断し、ロングを全停止。ショートのみ実施します。

**[B] 出来高フィルタ**  
直近5日の平均出来高 ÷ 過去20日の平均出来高 ≥ 0.7 の銘柄のみエントリー。出来高が細ってきた銘柄（流動性リスク）を除外します。

**[C] ファンディングレートブレンド（Bull時ショートのみ）**  
上昇相場でのショートは損しやすいため、FRが高い銘柄（買いポジションが積み上がっている）を優先してショートするよう補正します。

```
score_short = score - 0.3 × FR_zscore
```

---

### ③ 銘柄選定

全銘柄をスコアでランキングし、上位・下位を抜き出します。

| 相場 | ロング | ショート |
|------|--------|----------|
| Bull（BTC上昇中） | スコア上位15% | スコア下位10%（FR補正後） |
| Bear（BTC下落中） | なし（全停止） | 下位20%の中からさらに下位10% |

---

### ④ ポジションサイジング（ATRベース）

ボラティリティが高い銘柄ほど小口にするリスク管理です。

```
発注数量 = 運用資金 × リスク係数 / ATR（平均真値幅）
```

ATRが大きい（値動きが荒い）銘柄は少なく、ATRが小さい（値動きが穏やか）銘柄は多く建てることで、各ポジションのリスクを均一に保ちます。

また、自身のポートフォリオのパフォーマンスに応じてレバレッジを自動調整します。

| パフォーマンス | レバレッジ |
|----------------|-----------|
| 好調（移動平均を上回る） | 2.5倍 |
| 不調（移動平均を下回る） | 0.5倍に抑制 |

---

### ⑤ 注文執行

取引コストを下げるため、2段階の執行アルゴリズムを採用しています。

```
まず「指値注文（メイカー）」で約定を待つ
        ↓ タイムアウトしたら
「成行に近い指値（テイカー）」に切り替えて即時約定
```

メイカー注文は手数料が安いため、できるだけメイカーで約定させます。

---

### ⑥ 4トランシェ方式

ポートフォリオを4等分し、水・木・土・月曜日の23時に1/4ずつリバランスします。全ポジションを一度に入れ替えると特定の曜日の相場に依存してしまうため、時間分散によってリスクを低減します。

## 工夫した点

### 検証の厳密さ
- 「検証に使っていない期間のデータ」でも成立するかを必ず確認
- 複数のフィルタ条件の効果をそれぞれ個別に検証（アブレーション研究）
- 検証期間と実運用期間を明確に分離し、後出しの最適化をしない

## 学び

**バックテスト（事前検証）で良い結果が出ても、実運用で同じ結果が出るとは限らない。**

事前検証では高い指標を記録していたものの、実際の運用では想定を下回りました。そこで実際の取引記録をデータとして分析し、原因を以下のように特定しました。

| 原因 | 内容 |
|------|------|
| 相場環境への依存 | 特定の市場環境（下落相場）では戦略の一部がそもそも機能しない |
| 実際の取引コスト | 理論上の利益が手数料・スリッページで削られるケースがあった |
| 検証データへの過剰適合 | 検証期間特有の動きに最適化されすぎていた可能性 |

「数字を鵜呑みにしない」「うまくいかないときほどデータに立ち返る」という姿勢を、実体験として獲得しました。
